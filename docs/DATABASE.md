## 1. Design principles

**UUIDs as primary keys.** Every table uses `uuid` as its PK, generated by `gen_random_uuid()`. This avoids sequential ID enumeration in URLs and makes future database sharding straightforward.

**Soft deletes on user data only.** The `users` table has a `deleted_at` column. All other tables use hard deletes. User data must be recoverable for GDPR right-of-erasure workflows — a soft delete allows a 30-day recovery window before permanent purge.

**Prices stored in pence (integers).** All monetary values are stored as integers in the smallest currency unit (pence for GBP). This avoids floating-point rounding errors entirely. Display conversion happens in the application layer.

**Geography columns for spatial queries.** The `gyms` and `parks` tables use PostGIS `geography` type (not `geometry`) for the location column. `geography` operates in real-world metres and handles the curvature of the earth correctly — critical for accurate distance queries across cities.

**Cache timestamps on external data.** Any table that stores data fetched from a third-party API (`gyms`, `routes`, `community_groups`) includes a `last_fetched_at` or `cache_expires_at` column. The application checks this before deciding whether to re-fetch.

**`city_slug` as a lightweight city key.** Rather than a separate cities table (premature), city identity is represented as a normalised slug string (e.g. `london-gb`, `berlin-de`, `tokyo-jp`). This is sufficient for v1 and trivially migrated to a relational cities table when needed.

## 2. Extensions

These PostgreSQL extensions must be enabled before running migrations.

```sql
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";    -- gen_random_uuid()

```

In Supabase, PostGIS is enabled by default. `uuid-ossp` can be enabled from the Supabase dashboard under Database → Extensions.

## 3. Schema — core tables

### `users`

The auth record. Thin by design — we store only what Clerk does not already own. The `clerk_id` is the foreign key that links our database to Clerk's user system.

| Column | Type | Constraints | Notes |
|--------|------|-------------|-------|
| `id` | `uuid` | PK, default `gen_random_uuid()` | Internal ID |
| `clerk_id` | `text` | UNIQUE NOT NULL | Clerk's user ID (prefix: `user_`) |
| `email` | `text` | UNIQUE NOT NULL | Synced from Clerk on first login |
| `created_at` | `timestamptz` | NOT NULL, default `now()` | |
| `updated_at` | `timestamptz` | NOT NULL, default `now()` | Updated via trigger |
| `deleted_at` | `timestamptz` | nullable | Soft delete — null means active |

---

### `user_profiles`

The fitness profile. One-to-one with `users`. Contains everything that drives the match engine.

| Column | Type | Constraints | Notes |
|--------|------|-------------|-------|
| `id` | `uuid` | PK | |
| `user_id` | `uuid` | FK → users.id, UNIQUE NOT NULL | One profile per user |
| `training_style` | `text` | NOT NULL | Enum: `strength`, `calisthenics`, `cardio`, `crossfit`, `yoga`, `mixed` |
| `equipment_needs` | `text[]` | NOT NULL, default `{}` | Array: `free_weights`, `barbells`, `machines`, `bodyweight`, `pool` |
| `budget_range` | `text` | NOT NULL | Enum: `under_5`, `5_to_10`, `10_to_20`, `over_20` (GBP per day) |
| `max_distance_minutes` | `int` | NOT NULL, default `20` | Walking minutes acceptable |
| `environment_pref` | `text` | NOT NULL, default `both` | Enum: `indoor`, `outdoor`, `both` |
| `stay_length_pref` | `text` | NOT NULL, default `flexible` | Enum: `day`, `2_to_3_days`, `weekly`, `flexible` |
| `updated_at` | `timestamptz` | NOT NULL, default `now()` | |

**Why enums as text?** PostgreSQL native enums are painful to migrate — adding a value requires a table lock. Storing as constrained `text` with application-layer validation gives flexibility with negligible performance cost at this scale.

---

### `gyms`

Cached gym records sourced from Google Places. The `places_id` is the Google Places unique identifier — it's the key we use to avoid duplicates when re-fetching.

| Column | Type | Constraints | Notes |
|--------|------|-------------|-------|
| `id` | `uuid` | PK | |
| `places_id` | `text` | UNIQUE NOT NULL | Google Places ID (e.g. `ChIJ...`) |
| `name` | `text` | NOT NULL | |
| `address` | `text` | NOT NULL | Formatted address from Places API |
| `lat` | `float8` | NOT NULL | Stored separately for Prisma access |
| `lng` | `float8` | NOT NULL | Stored separately for Prisma access |
| `location` | `geography(Point, 4326)` | NOT NULL | PostGIS point — used for spatial queries |
| `opening_hours` | `jsonb` | nullable | Raw hours object from Places API |
| `equipment_tags` | `text[]` | NOT NULL, default `{}` | Community-sourced: `free_weights`, `barbells`, `rings`, etc. |
| `rating` | `float4` | nullable | Google Places rating (1.0–5.0) |
| `rating_count` | `int` | nullable | Number of Google reviews |
| `last_fetched_at` | `timestamptz` | NOT NULL, default `now()` | Used to decide cache validity |
| `created_at` | `timestamptz` | NOT NULL, default `now()` | |

**Note on `location` vs `lat`/`lng`:** Both exist because Prisma does not natively support PostGIS types — `lat` and `lng` are the values Prisma reads and writes, while `location` is kept in sync via a PostgreSQL trigger and is used exclusively by raw SQL spatial queries.

### `price_reports`

Crowdsourced day pass and weekly pricing. The single biggest data gap in the product — this table is how we fill it over time.

| Column | Type | Constraints | Notes |
|--------|------|-------------|-------|
| `id` | `uuid` | PK | |
| `gym_id` | `uuid` | FK → gyms.id NOT NULL | Which gym |
| `reported_by` | `uuid` | FK → users.id NOT NULL | Who submitted it |
| `day_pass_pence` | `int` | nullable | Day pass price in pence (null = not known) |
| `weekly_pence` | `int` | nullable | Weekly price in pence |
| `monthly_pence` | `int` | nullable | Monthly price in pence |
| `verified` | `bool` | NOT NULL, default `false` | Admin or community-verified flag |
| `visited_at` | `date` | NOT NULL | When the user actually visited |
| `created_at` | `timestamptz` | NOT NULL, default `now()` | |

The application uses the most recent `verified = true` price report for display. Unverified reports are shown with a "community estimate" label.

### `saved_gyms`

User bookmarks. Simple join table.

| Column | Type | Constraints | Notes |
|--------|------|-------------|-------|
| `id` | `uuid` | PK | |
| `user_id` | `uuid` | FK → users.id NOT NULL | |
| `gym_id` | `uuid` | FK → gyms.id NOT NULL | |
| `saved_at` | `timestamptz` | NOT NULL, default `now()` | |

Unique constraint on `(user_id, gym_id)` prevents duplicate saves.

---

### `gym_access`

The access passport record. Created when a user taps "I'm going here" on a gym card. In v1 this drives the cancellation reminder system. In v2 it becomes the foundation for brokered payment and auto-cancellation.

This is the table that transforms FitRoam from a discovery tool into an access management tool. Every row represents one user's decision to access one gym for a defined period.

| Column | Type | Constraints | Notes |
|--------|------|-------------|-------|
| `id` | `uuid` | PK | |
| `user_id` | `uuid` | FK → users.id NOT NULL | |
| `gym_id` | `uuid` | FK → gyms.id NOT NULL | |
| `city_slug` | `text` | NOT NULL | Denormalised for fast history queries |
| `access_type` | `text` | NOT NULL | Enum: `day_pass`, `weekly`, `monthly`, `unknown` |
| `price_paid_pence` | `int` | nullable | What the user paid — null if not tracked in v1 |
| `activated_at` | `timestamptz` | NOT NULL, default `now()` | When user tapped "I'm going here" |
| `expected_end_date` | `date` | NOT NULL | User's recorded departure date — drives reminder |
| `reminder_sent_at` | `timestamptz` | nullable | When the cancellation reminder was sent |
| `cancelled_at` | `timestamptz` | nullable | If user confirmed cancellation (v2) |
| `status` | `text` | NOT NULL, default `'active'` | Enum: `active`, `reminder_sent`, `cancelled`, `expired` |
| `notes` | `text` | nullable | Free text — e.g. "login saved in notes app" |
| `created_at` | `timestamptz` | NOT NULL, default `now()` | |

**Key behaviour:** A background job runs daily checking for rows where `expected_end_date = tomorrow` and `reminder_sent_at IS NULL`. For each match it sends a push notification and sets `reminder_sent_at`. This is the entire v1 passport mechanism — simple, reliable, high value.

**Why `city_slug` is denormalised here:** The access history screen groups entries by city. Joining through `gyms` on every history query adds unnecessary complexity. Storing `city_slug` directly makes the query a single indexed scan.

## 6. Indexes

These indexes are critical for performance. Without them, spatial queries degrade to full table scans.

```sql
-- Spatial indexes (PostGIS GIST) — essential for ST_DWithin queries
CREATE INDEX idx_gyms_location      ON gyms      USING GIST (location);
CREATE INDEX idx_parks_location     ON parks     USING GIST (location);

-- Lookup indexes
CREATE INDEX idx_gyms_places_id     ON gyms      (places_id);
CREATE INDEX idx_routes_city_slug   ON routes    (city_slug);
CREATE INDEX idx_parks_osm_id       ON parks     (osm_id);
CREATE INDEX idx_community_city     ON community_groups (city_slug);
CREATE INDEX idx_price_gym_id       ON price_reports (gym_id);
CREATE INDEX idx_saved_user_id      ON saved_gyms (user_id);
CREATE INDEX idx_trips_user_id      ON trips (user_id);

-- Access passport indexes
-- The reminder job queries by expected_end_date + reminder_sent_at daily — this index is critical
CREATE INDEX idx_access_reminder    ON gym_access (expected_end_date, reminder_sent_at)
  WHERE reminder_sent_at IS NULL;
CREATE INDEX idx_access_user_id     ON gym_access (user_id);

-- Partial index — only active users
CREATE INDEX idx_users_active       ON users (id) WHERE deleted_at IS NULL;
```

## 7. Prisma schema file

This is the actual `schema.prisma` file. Copy it to `packages/api/prisma/schema.prisma` in your repo.

```prisma
generator client {
  provider        = "prisma-client-js"
  previewFeatures = ["postgresqlExtensions"]
}

datasource db {
  provider   = "postgresql"
  url        = env("DATABASE_URL")
  extensions = [uuidOssp(map: "uuid-ossp"), postgis]
}

model User {
  id        String    @id @default(dbgenerated("gen_random_uuid()")) @db.Uuid
  clerkId   String    @unique @map("clerk_id")
  email     String    @unique
  createdAt DateTime  @default(now()) @map("created_at") @db.Timestamptz
  updatedAt DateTime  @updatedAt @map("updated_at") @db.Timestamptz
  deletedAt DateTime? @map("deleted_at") @db.Timestamptz

  profile       UserProfile?
  priceReports  PriceReport[]
  savedGyms     SavedGym[]
  trips         Trip[]

  @@map("users")
}

model UserProfile {
  id                  String   @id @default(dbgenerated("gen_random_uuid()")) @db.Uuid
  userId              String   @unique @map("user_id") @db.Uuid
  trainingStyle       String   @map("training_style")
  equipmentNeeds      String[] @map("equipment_needs")
  budgetRange         String   @map("budget_range")
  maxDistanceMinutes  Int      @default(20) @map("max_distance_minutes")
  environmentPref     String   @default("both") @map("environment_pref")
  stayLengthPref      String   @default("flexible") @map("stay_length_pref")
  updatedAt           DateTime @updatedAt @map("updated_at") @db.Timestamptz

  user User @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@map("user_profiles")
}

model Gym {
  id            String    @id @default(dbgenerated("gen_random_uuid()")) @db.Uuid
  placesId      String    @unique @map("places_id")
  name          String
  address       String
  lat           Float
  lng           Float
  openingHours  Json?     @map("opening_hours")
  equipmentTags String[]  @default([]) @map("equipment_tags")
  rating        Float?
  ratingCount   Int?      @map("rating_count")
  lastFetchedAt DateTime  @default(now()) @map("last_fetched_at") @db.Timestamptz
  createdAt     DateTime  @default(now()) @map("created_at") @db.Timestamptz

  priceReports PriceReport[]
  savedBy      SavedGym[]
  tripGyms     TripGym[]

  @@map("gyms")
}

model PriceReport {
  id            String   @id @default(dbgenerated("gen_random_uuid()")) @db.Uuid

  reportedBy    String   @map("reported_by") @db.Uuid
  dayPassPence  Int?     @map("day_pass_pence")
  weeklyPence   Int?     @map("weekly_pence")
  monthlyPence  Int?     @map("monthly_pence")
  verified      Boolean  @default(false)
  visitedAt     DateTime @map("visited_at") @db.Date
  createdAt     DateTime @default(now()) @map("created_at") @db.Timestamptz

  gym          Gym  @relation(fields: [gymId], references: [id])
  reportedByUser User @relation(fields: [reportedBy], references: [id])

  @@map("price_reports")

model SavedGym {
  id      String   @id @default(dbgenerated("gen_random_uuid()")) @db.Uuid
  userId  String   @map("user_id") @db.Uuid
  gymId   String   @map("gym_id") @db.Uuid

  user User @relation(fields: [userId], references: [id])
  gym  Gym  @relation(fields: [gymId], references: [id])

model GymAccess {
  id              String    @id @default(dbgenerated("gen_random_uuid()")) @db.Uuid
  userId          String    @map("user_id") @db.Uuid
  gymId           String    @map("gym_id") @db.Uuid
  citySlug        String    @map("city_slug")
  accessType      String    @map("access_type")

  createdAt       DateTime  @default(now()) @map("created_at") @db.Timestamptz

  citySlug      String  @map("city_slug")

  description   String?
  distanceKm    Float   @map("distance_km")
  terrainType   String  @map("terrain_type")
  surfaceType   String  @map("surface_type")

model CommunityGroup {
  id                  String    @id @default(dbgenerated("gen_random_uuid()")) @db.Uuid

  name                String
  type                String
  meetupId            String?   @map("meetup_id")
```

## 8. Migration strategy

Prisma manages all schema migrations via `prisma migrate`.

```bash
# Create a new migration (run whenever schema.prisma changes)
npx prisma migrate dev --name <description>

# Apply migrations in production
npx prisma migrate deploy

# Inspect the current database state
npx prisma db pull

# Open Prisma Studio (visual DB browser — useful during development)
npx prisma studio
```

**Rules:**
- Never edit a migration file after it has been applied to any environment (dev, staging, production).
- Every migration file is committed to Git. The migration history is the ground truth for the database state.
- Run `npx prisma generate` after any schema change to regenerate the TypeScript client.
- The PostGIS `location` column and its GIST index are created via a raw SQL migration, not managed by Prisma (Prisma does not support PostGIS types natively). That migration lives at `prisma/migrations/manual/001_add_postgis_columns.sql`.
